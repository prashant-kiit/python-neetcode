# Thread Safety in Python

## How Thread Safety is Ensured

Even though the **GIL** allows only one thread to execute Python bytecode at a time, **thread safety is NOT guaranteed automatically**. The GIL can switch between threads *mid-operation* (e.g., between bytecode instructions), so operations like `x += 1` (read-modify-write) are **not atomic** and can still cause race conditions.

**Ways Python ensures thread safety:**

1. **Synchronization primitives** from the `threading` module — `Lock`, `RLock`, `Semaphore`, `Event`, `Condition` — used to manually protect shared resources.
2. **Atomic operations on certain built-ins** — simple operations like `list.append()` or dict assignment are atomic *at the bytecode level* due to the GIL, but compound operations (`x += 1`, `if x: x = y`) are **not**.
3. **Thread-safe data structures** — `queue.Queue` is internally lock-protected, safe to use across threads without manual locking.
4. **Avoiding shared state** — using `asyncio` or `multiprocessing` (separate memory) sidesteps the problem entirely.

```python
import threading

counter = 0
lock = threading.Lock()

def increment():
    global counter
    with lock:        # ensures only one thread modifies counter at a time
        counter += 1
```

## Mutex (`Lock`) vs `RLock` vs `Semaphore` vs `Event`

### 1. `Lock` (Mutex)
- **Binary lock** — only 2 states: locked / unlocked.
- Only **one thread** can hold it at a time.
- **Not reentrant** — if the *same* thread tries to acquire it again while already holding it → **deadlock**.

```python
lock = threading.Lock()
lock.acquire()
# critical section
lock.release()
```

**Use case:** protecting a single shared resource (counter, shared list) from concurrent modification.

---

### 2. `RLock` (Reentrant Lock)
- Same as `Lock`, but the **same thread can acquire it multiple times** without deadlocking itself.
- Internally tracks **owner thread + acquisition count** — must be released the same number of times it was acquired.

```python
rlock = threading.RLock()

def outer():
    with rlock:
        inner()   # same thread re-acquiring — works fine with RLock, would deadlock with Lock

def inner():
    with rlock:
        pass
```

**Use case:** recursive functions or nested methods that both need the same lock.

---

### 3. `Semaphore`
- A **counter-based lock** — allows **N threads** to access a resource simultaneously (not just 1).
- Each `acquire()` decrements the counter, `release()` increments it. Blocks when counter hits 0.

```python
sem = threading.Semaphore(3)   # max 3 threads allowed at once

def access_resource():
    with sem:
        # up to 3 threads can be here concurrently
        pass
```

**Use case:** limiting concurrent access — e.g., max 3 DB connections, rate-limiting concurrent API calls.

*(Special case: `Semaphore(1)` behaves like a `Lock`.)*

---

### 4. `Event`
- **Not a lock at all** — it's a **signaling mechanism** between threads.
- One thread **sets** a flag, other threads **wait** for that flag to become true.

```python
event = threading.Event()

def waiter():
    event.wait()      # blocks until event.set() is called
    print("Go!")

def setter():
    event.set()        # unblocks all waiting threads
```

**Use case:** one thread signaling "I'm done" / "data is ready" to other waiting threads — no mutual exclusion involved, just coordination.

---

## Quick Comparison Table

| Primitive | Purpose | Allows Multiple Holders? | Reentrant? |
|---|---|---|---|
| `Lock` | Mutual exclusion | No (1 only) | No |
| `RLock` | Mutual exclusion (same thread safe) | No (1 only) | Yes |
| `Semaphore` | Limit concurrent access | Yes (N) | No |
| `Event` | Signal/notify between threads | N/A (not exclusion) | N/A |

**One-liner:** "Lock is a basic mutex for exclusive access; RLock is the same but lets the owning thread re-acquire it without deadlocking itself; Semaphore extends the idea to allow N concurrent holders instead of just one; and Event isn't about exclusion at all — it's a simple flag-based signal for one thread to notify others."
