`__dict__` and `__slots__` control **how Python stores an object's attributes**.

## `__dict__` (Default)

By default, every instance has a `__dict__`, which is a dictionary mapping attribute names to values.

```python
class Employee:
    pass

e = Employee()
e.name = "Alice"
e.age = 30

print(e.__dict__)
```

Output:

```python
{'name': 'Alice', 'age': 30}
```

Internally:

```
e
 └── __dict__
      ├── "name" → "Alice"
      └── "age"  → 30
```

### Advantages

* Can add attributes dynamically.
* Easy introspection.
* Works well with serialization libraries.

```python
e.salary = 100000

print(e.__dict__)
```

```
{
    'name': 'Alice',
    'age': 30,
    'salary': 100000
}
```

---

## `__slots__`

`__slots__` tells Python:

> "Don't create a `__dict__`. Only allow these attributes."

```python
class Employee:
    __slots__ = ("name", "age")

e = Employee()
e.name = "Alice"
e.age = 30

print(e.name)
```

Trying to create a new attribute:

```python
e.salary = 100000
```

raises

```
AttributeError:
'Employee' object has no attribute 'salary'
```

---

## Memory layout

### Without `__slots__`

```
Employee object
      |
      +-------> __dict__
                    |
      -------------------------------
      | "name" | "age" | "salary" |
      -------------------------------
```

Extra memory is needed for the dictionary itself.

---

### With `__slots__`

```
Employee object
------------------------
| slot0 | slot1 |
------------------------
  name     age
```

No dictionary lookup.

The values live directly in fixed locations.

---

# Performance

Attribute lookup:

Without slots:

```
obj
 ↓
__dict__
 ↓
hash("name")
 ↓
dictionary lookup
 ↓
value
```

With slots:

```
obj
 ↓
offset 0
 ↓
value
```

It's slightly faster.

The biggest gain is **memory**, not speed.

---

# Memory comparison

Suppose

```python
class A:
    pass

class B:
    __slots__ = ("x", "y")
```

Create one million objects.

Without slots:

```
1,000,000 × object
1,000,000 × dict

lots of memory
```

With slots:

```
1,000,000 × object

no per-instance dict

much less memory
```

Savings are often **20–50% or more**, depending on the number of attributes and the Python version.

```
True
```

# If you still want a `__dict__`

You can explicitly include it in `__slots__`:

```python
class Employee:
    __slots__ = ("name", "__dict__")
```

Now:

```python
e = Employee()

e.name = "Alice"
e.salary = 100
```

works because dynamic attributes go into `__dict__`.

---

# Inheritance

```python
class Person:
    __slots__ = ("name",)

class Employee(Person):
    __slots__ = ("salary",)
```

The child gets slots for:

```
name
salary
```

If a subclass **doesn't** define `__slots__`, Python will create a `__dict__` for the subclass instances, allowing dynamic attributes again.

---

# When to use `__slots__`

Use it when:

* You create **millions of objects**.
* Objects have a **fixed set of attributes**.
* Memory usage matters (e.g., game engines, simulations, parsers, data models).

Avoid it when:

* You need to add attributes dynamically.
* You rely on libraries that expect `__dict__`.
* Memory savings are negligible for your application.

---

# Example

Without slots:

```python
class Point:
    pass

p = Point()
p.x = 1
p.y = 2
p.z = 3  # allowed
```

With slots:

```python
class Point:
    __slots__ = ("x", "y")

p = Point()
p.x = 1
p.y = 2

p.z = 3
```

```
AttributeError
```

# Summary

| Feature                 | `__dict__`              | `__slots__`                           |
| ----------------------- | ----------------------- | ------------------------------------- |
| Dynamic attributes      | ✅ Yes                   | ❌ No (unless `__dict__` is included)  |
| Per-instance dictionary | ✅ Yes                   | ❌ No                                  |
| Memory usage            | Higher                  | Lower                                 |
| Attribute lookup        | Dictionary lookup       | Fixed slot lookup (slightly faster)   |
| Flexibility             | High                    | Lower                                 |
| Best for                | General-purpose classes | Large numbers of fixed-layout objects |

In interviews, the key takeaway is:

* **`__dict__`** stores instance attributes in a hash table, providing flexibility at the cost of additional memory.
* **`__slots__`** replaces the per-instance dictionary with a fixed set of attribute slots, reducing memory usage and slightly speeding up attribute access, but restricting instances to predefined attributes unless `__dict__` is explicitly included.
